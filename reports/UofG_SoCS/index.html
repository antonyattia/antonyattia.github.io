This site is about my Summer 2016 work term at the School of Computer Science in the University of Guelph. I will be outlining my experiences with the SoCS, as well as what I've learned and my reflections.

The University of Guelph was founded in 1964, but its origins can be traced ninety years prior to its founding; in 1874. It has since been one of the top comprehensive universities in Canada.

The School of Computer Science at the University of Guelph was founded in 1971. SoCS has a plethora of talented faculty and staff, some of which I had the pleasure to work with (Greg Klotz, Kyle Johnston and Judi McCuaig).

I've been a student at the University of Guelph for two full years so far, now starting my third consecutive year.

Being assigned the role of Undergraduate Research Assistant, I was assigned to do mainly two things:

  • Create a program (Projection/Hourglass) that analyzes course and student data, calculates relevant statistics for it, and also create estimations for the future statistics.
  • Refactor the back-end and improve the web front-end for Moss (Measure of Software Similarity).

Projection/Hourglass was targeted specifically towards School of Computer Science faculty and staff use. Moss on the other hand, was more broad and could be used by any instructor for CIS courses (including professors and authorized TAs).


Projection/Hourglass

The data analysis program (codenamed "Projection" or "Hourglass") was written primarily with Python (including Python modules/packages available online) and some SQL (specifically MySQL). The program was required to read in .csv (comma seperated values) files containing the course data (specifically course information and information on the students attending the course.), and store the data read. The data is then used to calculate statistics (such as grade averages, drop/fail rate, course overlap rates, etc.).

Projection/Hourglass made significant use of data structures (mostly hash tables), object-oriented programming (classes/objects made for each course and student), and command-line arguments (depending on the argument given, functionality would change accordingly). It can run in both the command-line and a graphical user interface. To give you an idea of the wide range of functionality of the program, here some command-line arguments flags that can be used:

  • No command-line argument (only .csv files or directories inputted)
    • The program will print out the statistics/comparisons of all the courses listed in all the inputted files to the terminal and to a new .csv file for output.
  • -GUI
    • Enables the graphical user interface component of the program. Rather than using and printing to the terminal, you can select your files through the GUI and also see the output in a window.
  • -program
    • The program will run course comparisons only for courses within the specified program (according to the coded program outlines).
    • Example: '-program BCH.CS' would enable course comparisons (student overlap) only for the mandatory courses in the Bachelor of Computing - Computer Science program.
  • -projection
    • The program will estimate projections for courses using the formula:
    • [W16_CIS*2500 num_students] = [F15_CIS*1500 num_students] * [F14_CIS*1500 --> W15_CIS*2500 avg_overlap] + [W15_CIS*2500 num_drop, num_fail]

The primary use of Projection/Hourglass is to look at the course overlaps (using the -program flag) and course projections (using the -projection flag). This is useful because it helps the School of Computer Science to decide how many slots to allocate for course sections, as well as see the overall progress of students and how they advance through the program. This can be insightful when considering course criteria and content.


Moss

Moss is a software system developed by Stanford to detect plagiarism, particularly in programming assignments. It supports a wide variety of languages, and is known to be quite accurate and effective.

The front-end consisted of an upload page, only accessible to instructors. This upload page required the professor/TA to:

  1. Select the appropriate course.
  2. Provide a unique one-word name for the assignment.
  3. Select the language said assignment was written in.
  4. Upload the archive file containing the student work to be analyzed.
  5. Optionally upload an archive file containing files given to the students by the instructor.
  6. Optionally input any comments they would like to mention to the academic integrity team.

After a successfull submission, the back-end then handles and processes the assignment archive according to the other settings the instructor set (course, language, name).

The back-end is written in Python, and the web front-end in HTML/CSS. The back-end had to be changed to allow for automatic submission reorganization and archiving, whereas the front-end needed to be changed to improve the user interface and general ease-of-use.


Goals

Since this was my first 4-month work term, I made sure to make my goals straight-forward so they would not be unattainable:

  • Become more proficient with Python and gain more experience with SQL.
    • Having prior experience with Python (from previous courses and projects), I was happy to see that I'd be using Python once again. It is a very clean language, where you can get a lot done with not a lot of code. Most importantly, it's very readable, so other people reading my code will not have a hard time trying to make sense of it (given there are comments and documentation as well, of course). This is important because it's more than likely that the code I worked on will be worked on even further by someone else.
    • I was interested in gaining more experience with SQL, since it is almost always useful and advantageous to know how to use databases (both relational and non-relational). Initially, Projection/Hourglass utilized an in-memory database (meaning all data was stored as it was read from the file into memory), which works perfectly well, but for long-term purposes (i.e. utilizing a large number of course files, each containing a large number of students), a relational database would prove to be useful since the files would not have to be read everytime, but rather the data would be stored in the database as soon as it is read.
  • Make sure my time and work were organized in a productive and efficient manner.
    • Since this wasn't a conventional job (i.e. working in an office with co-workers in a software development team), but rather a research position, I was working alone most of the time. Because I was working alone, I was generally on my own schedule. I wanted to make sure that the schedule I set for myself was efficient both in terms of time management and splitting up the work.
  • Talk more frequently/freely with my supervisors and asking for help/questions when need-be, as well as providing regular status/progress updates.
    • I prefer to refrain from asking for help unless it's absolutely necessary. This is because not only do I not want to bother my peers with a bombardment of questions, but also because I feel as if I won't learn as much if someone is always holding my hand. Although this is true to an extent, it is important to ask questions and to regularly update your co-workers/supervisors on your progress.
    • To put it simply, if you're spending time on something and the progress made isn't at least somewhat proportional to the amount of time spent, it's time to ask questions.

Working at the School of Computer Science has been a significant and relevant learning experience. The skills I've learned during my time will prove to be invaluable, not only for my professional career but also my academic one. My problem solving skills have improved after significant use of Python and some use of SQL, and my communication and disciplinary skills have also improved, with frequent communication with my supervisor and partners.

My experience with the School of Computer Science and the University of Guelph as an employer in general has been superb and eye-opening, so I'd like to thank Greg Klotz and Kyle Johnston for their guidance and supervision, and I'd also like to thank Laura Gatto, Judi McCuag, and Pascal Matsakis for providing me with this great opportunity to learn and advance my skills, as well as giving back to the school at the same time.
